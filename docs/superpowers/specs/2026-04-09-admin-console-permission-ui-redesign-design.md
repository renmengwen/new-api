# 运营后台菜单与权限管理重构设计

日期：2026-04-09

## 目标

本次重构解决 4 类问题：

1. 运营后台当前 UI 过于临时，页面风格与现有后台主页面不一致。
2. 代理商管理、权限管理和用户管理被耦合在同一入口下，信息架构不清晰。
3. 当前权限页只能“绑定模板”，不能满足按用户或代理商做细颗粒度增删权限。
4. 菜单可见性、动作权限和数据范围没有统一成一套稳定模型，后续扩展容易返工。

本次改造后的目标是：

- 运营后台一级菜单独立化。
- 页面视觉风格对齐现有“渠道管理”页的后台表格风格。
- 权限体系支持“模板快速赋权 + 用户级颗粒度覆盖”。
- 菜单可见性、动作权限、数据范围三层能力统一建模。

## 非目标

- 本次不处理第二阶段统计、模型、导出等新业务页面。
- 本次不重做现有全站设计系统，只在运营后台相关页面统一样式。
- 本次不废弃已有权限模板表结构，而是在此基础上扩展。

## 信息架构

运营后台拆为 5 个一级菜单：

1. 用户管理
2. 代理商管理
3. 权限模板管理
4. 用户权限管理
5. 额度流水

职责边界如下：

### 用户管理

- 面向普通用户。
- 支持列表、查询、详情、启停、单用户调额、批量调额。
- 不承载后台权限配置。

### 代理商管理

- 面向代理商账号。
- 支持列表、查询、创建、编辑、启停、额度摘要查看。
- 不承载模板编辑和权限矩阵编辑。

### 权限模板管理

- 面向权限模板本身。
- 支持模板列表、创建、编辑、启停。
- 支持按资源和动作配置颗粒度权限。
- 支持配置菜单可见项和默认数据范围。

### 用户权限管理

- 面向账号级最终权限。
- 可管理对象包括管理员、代理商、普通用户。
- 支持：
  - 绑定模板
  - 清空模板
  - 对单个账号做颗粒度权限覆盖
  - 单独配置菜单可见性
  - 单独配置数据范围

### 额度流水

- 保持账本查询能力。
- 仅负责查询和筛选，不承担调额入口聚合。

## 权限模型

采用“模板 + 用户级覆盖”的最小返工方案。

### 1. 模板权限

权限模板继续作为“快速初始化权限基线”。

模板中维护：

- 适用对象类型：`admin` / `agent` / `end_user`
- 菜单可见性
- 资源动作权限
- 默认数据范围

### 2. 用户级覆盖

新增用户级覆盖层，用于对模板结果做增删。

最终权限计算规则：

- `root` 用户始终全量放行。
- 其他账号的最终权限：
  - `最终权限 = 模板权限 + 用户显式允许 - 用户显式禁止`

用户级覆盖分为三类：

1. 动作权限覆盖
2. 菜单可见性覆盖
3. 数据范围覆盖

### 3. 菜单、动作、数据范围解耦

三者必须分开建模：

- 菜单可见性：是否能看到菜单。
- 动作权限：是否能进入页面、查询、新增、编辑、启停、调额。
- 数据范围：能看到或操作哪些对象。

这允许以下场景成立：

- 代理商可以看到“用户权限管理”菜单。
- 代理商可以访问该页面。
- 但其数据范围是 `agent_only`。
- 最终只能管理自己绑定的用户。

### 4. 普通用户的后台入口

普通用户也允许出现在“用户权限管理”里，但默认无后台权限。

默认状态：

- 不绑定模板
- 无后台菜单
- 无后台动作权限

如果超管单独给普通用户分配菜单、动作和数据范围，则其可获得对应后台入口。

## 数据模型

在现有 `permission_profiles`、`permission_profile_items`、`user_permission_bindings` 基础上扩展。

### 新增表：`user_permission_overrides`

用途：对单个用户做动作权限覆盖。

建议字段：

- `id`
- `user_id`
- `resource_key`
- `action_key`
- `effect`：`allow` / `deny`
- `created_at`
- `updated_at`

### 新增表：`user_menu_overrides`

用途：对单个用户做菜单可见性覆盖。

建议字段：

- `id`
- `user_id`
- `section_key`
- `module_key`
- `effect`：`show` / `hide`
- `created_at`
- `updated_at`

### 新增表：`user_data_scope_overrides`

用途：对单个用户做数据范围覆盖。

建议字段：

- `id`
- `user_id`
- `resource_key`
- `scope_type`
- `scope_value_json`
- `created_at`
- `updated_at`

说明：

- 如果希望进一步减少表数量，也可以把菜单和数据范围覆盖存入一个通用覆盖表。
- 但从可读性和后续维护看，三类覆盖拆表更稳。

## 后端设计

### 权限模板管理

新增能力：

- 模板列表
- 模板详情
- 创建模板
- 更新模板
- 模板动作权限项编辑
- 模板菜单项编辑
- 模板默认数据范围编辑

现有 `permission_profiles` 和 `permission_profile_items` 继续复用。

### 用户权限管理

新增能力：

- 账号列表查询
- 查看账号当前模板
- 查看账号最终权限
- 绑定模板
- 清空模板
- 保存用户级动作权限覆盖
- 保存用户级菜单可见性覆盖
- 保存用户级数据范围覆盖

### 统一权限计算服务

新增或扩展统一服务，负责：

1. 读取模板权限
2. 读取用户级覆盖
3. 合并出最终菜单可见性
4. 合并出最终动作权限
5. 合并出最终数据范围

要求：

- 控制器和前端都不直接拼权限。
- 所有 `/api/admin/*` 页面与接口统一调用该服务。

## 前端设计

### 风格

运营后台页面风格对齐“渠道管理”页，遵循以下约束：

- 使用表格主视图 + 顶部筛选工具条。
- 主内容不再塞入过重的嵌套卡片。
- 弹窗统一为后台表单风格，不使用现在这种简陋的竖排输入堆叠。
- 详情、编辑、调额、权限配置弹窗采用统一尺寸、统一标题区、统一底部操作区。

### 用户管理

保留现有能力，但做 UI 重构：

- 表格对齐渠道管理风格
- 批量调额按钮放在工具条
- 调额弹窗改为更明确的表单布局
- 详情弹窗改为信息区块 + 额度摘要区块

### 代理商管理

新增“编辑代理商”按钮。

页面包含：

- 筛选区
- 列表
- 新增代理商弹窗
- 编辑代理商弹窗
- 详情弹窗

### 权限模板管理

新页面包含：

- 模板列表
- 模板适用对象类型筛选
- 新增/编辑模板弹窗
- 权限矩阵编辑区

权限矩阵至少分 3 块：

1. 菜单可见性
2. 动作权限
3. 默认数据范围

### 用户权限管理

新页面包含：

- 账号列表
- 按对象类型筛选：管理员 / 代理商 / 普通用户
- 当前模板列
- “配置权限”按钮

配置权限弹窗包含：

1. 模板绑定区
2. 菜单可见性覆盖区
3. 动作权限覆盖区
4. 数据范围覆盖区

每项覆盖建议使用三态交互：

- 继承模板
- 显式允许
- 显式禁止

这样能准确表达覆盖含义，避免只靠单个勾选框导致语义不清。

## 菜单权限规则

运营后台后续所有菜单统一遵循：

1. 菜单是否显示由最终菜单权限决定。
2. 菜单显示不代表有写权限。
3. 页面可打开不代表能看到全部数据。
4. 数据访问必须再经过服务端数据范围校验。

## 错误处理

### 后端

- 模板不匹配用户类型时拒绝保存。
- 代理商尝试操作超出数据范围的账号时拒绝保存。
- 用户级覆盖保存失败时，不得写入部分脏数据。

### 前端

- 列表页统一区分：
  - 加载失败
  - 无数据
  - 无匹配结果
- 配置弹窗保存失败时保留当前编辑内容，不自动关闭。
- 权限矩阵加载失败时提供重试入口。

## 测试策略

### 后端

新增测试覆盖：

- 模板权限与用户覆盖合并逻辑
- 菜单可见性覆盖
- 数据范围覆盖
- 代理商可见菜单但受数据范围限制
- 普通用户默认无权限，单独赋权后生效

### 前端

新增验证：

- 运营后台一级菜单显示正确
- 代理商管理页可编辑
- 权限模板管理页可新增/编辑模板
- 用户权限管理页可绑定模板和做用户级覆盖
- 代理商看到菜单但只能看到自己绑定用户的数据

## 实施顺序

1. 后端新增用户级覆盖表和统一权限合并逻辑
2. 后端新增权限模板管理接口和用户权限管理接口
3. 前端拆一级菜单
4. 前端重做权限模板管理页
5. 前端重做用户权限管理页
6. 前端统一收口用户管理、代理商管理和调额弹窗样式

## 风险与约束

### 风险

- 如果把菜单、动作、数据范围继续混在一层，会导致后续菜单扩展持续返工。
- 如果用户级覆盖不支持“显式禁止”，模板将无法被安全收缩。
- 如果代理商菜单显示逻辑只做前端判断，仍会留下越权风险。

### 约束

- 必须保持现有三库兼容。
- 必须继续使用后端统一权限服务，不允许前端自行拼最终权限。
- 不修改受保护的项目标识与版权信息。

## 结论

本次重构采用“模板 + 用户级覆盖”的方案，在最小返工前提下同时满足：

- 一键绑定模板
- 给不同管理员、代理商、普通用户配置不同颗粒度权限
- 按人控制菜单可见性
- 按人控制数据范围
- 将运营后台拆成清晰的一组一级菜单

该方案可直接作为下一步实现计划与开发基线。
