# Website Intro Design

Date: 2026-04-15

## Goal

产出一份面向外部客户、合作方和项目汇报场景的 Markdown 介绍稿，用于说明该网站和平台的产品能力。

这份介绍稿的核心目标不是讲解后台如何操作，而是清晰表达：

- 平台是什么
- 平台解决什么问题
- 平台为什么适合做统一 AI 接入与管理
- 管理员后台有哪些关键治理能力

其中，正文重点必须放在以下四个方向：

- 管理员功能
- 系统设置
- 渠道管理
- 分组管理

## Audience

目标读者包括：

- 需要快速理解平台能力的潜在客户
- 需要阅读项目介绍材料的合作方
- 需要用于方案汇报、立项汇报或交付说明的业务方与管理层

这意味着文案必须满足两个条件：

- 读起来像产品介绍，而不是开发文档
- 内容仍然必须与当前网站页面和代码实现保持一致

## Output

最终输出是一篇可以直接复用的 Markdown 文案。

文案特征：

- 适合直接发给客户或贴入项目汇报材料
- 适合在官网介绍页、方案文档、交付文档中复用
- 篇幅中等，不做逐项操作教程
- 用产品化语言总结能力，不下沉到字段级和按钮级说明

明确排除项：

- 不写成后台操作手册
- 不写成接口清单
- 不引入代码中不存在的规划型能力
- 不使用“点击这里、进入这里、保存这里”这类使用步骤

## Product Positioning

文案将把该平台定位为统一的大模型网关与管理平台，而不只是请求转发层。

对外表达主线：

- 统一接入多家 AI 服务提供商与模型能力
- 通过统一接口、统一网关和统一控制台降低接入复杂度
- 通过管理员后台实现渠道、分组、权限、计费、日志和系统配置的统一治理
- 支持从个人团队到企业级平台化运营的管理需求

## Source Basis

介绍稿的能力描述仅允许来自当前已存在的页面、导航结构、设置项和后端接口。

本次写作已确认的事实基础包括：

- 首页与 README 明确传达平台具备多模型、多供应商接入定位
- 管理后台侧边栏已包含渠道管理、订阅管理、模型管理、模型部署、兑换码管理、用户管理、管理员管理、代理商管理、权限模板管理、用户权限管理、额度流水、系统设置等模块
- 系统设置页已覆盖登录注册、OAuth/OIDC、Passkey、邮件、公告、服务地址、Worker、Turnstile、邮箱白名单、SSRF 防护等平台级配置
- 渠道管理页已支持渠道列表、搜索、筛选、启停、复制、测试、余额更新、批量删除、批量标签、标签模式、上游更新检测与处理等能力
- 分组能力已覆盖系统分组列表、分组倍率、用户可选分组、分组特殊倍率、分组特殊可用分组、自动分组 `auto`、默认使用自动分组、预填分组管理等内容

因此，最终文案可以强调“统一治理能力”，但不能把未落地的高级企业功能写成现有能力。

## Writing Strategy

最终成稿采用“官网介绍型 + 汇报材料型”的结合方式。

写法顺序：

1. 先说明产品定位和业务价值
2. 再用简洁的能力总览建立整体印象
3. 然后把管理员后台能力作为核心章节展开
4. 最后补充适用场景和总结

这样写的原因：

- 对外阅读时先讲价值，理解成本最低
- 汇报场景下先讲问题与解决方案，更容易建立说服力
- 管理员能力作为核心卖点，可以自然承接系统设置、渠道管理、分组管理等重点内容

## Structure

最终 Markdown 按以下结构组织：

### 1. 产品概述

说明平台是统一接入多家 AI 服务商的大模型网关与管理平台，兼顾接入、运营、权限、计费和监控。

### 2. 核心能力总览

用 4 到 6 条高度概括的平台能力建立整体认知，例如：

- 多模型与多供应商统一接入
- OpenAI 兼容风格的统一接口输出
- 管理员后台统一治理
- 渠道调度与多上游运维
- 分组与定价策略控制
- 面向运营与交付的可视化管理能力

### 3. 管理员后台能力

这一章作为正文重点入口，先总述管理员能够统一管理：

- 渠道
- 用户与管理员
- 权限与代理商
- 订阅与兑换码
- 日志与额度流水
- 系统级配置

这一章不拆到过细，而是起到“管理价值总览”的作用。

### 4. 系统设置

重点写平台级控制能力，而不是配置项罗列。

应强调的方向：

- 登录与注册方式控制
- GitHub、Discord、OIDC、Telegram、LinuxDO、微信等接入能力
- Passkey 与安全校验能力
- 邮件与通知基础设施配置
- 公告、首页内容、关于页面、页脚等站点级内容管理
- 服务地址、Worker、代理与安全访问策略
- SSRF 防护、邮箱域白名单、验证码等安全治理能力

这一章的价值表达重点是：平台可以被配置成适合不同组织和不同上线环境的运行形态。

### 5. 渠道管理

这是最重要的核心章节之一。

必须覆盖的能力点：

- 统一接入和维护多家模型供应商渠道
- 渠道的新增、编辑、复制、启用、禁用和删除
- 按名称、分组、模型、状态、类型进行检索和筛选
- 渠道标签化管理与批量操作
- 渠道测试、模型测试、余额更新和可用性维护
- 上游模型变更检测与同步处理
- 多渠道并行管理下的优先级、权重、状态控制

这一章的核心价值表达是：管理员可以把复杂的上游模型资源管理集中在一个控制台内完成。

### 6. 分组管理

这是第二个重点章节。

必须覆盖的能力点：

- 基于分组控制不同用户或令牌的使用范围
- 通过分组倍率实现差异化计费或资源策略
- 定义用户可选分组和分组说明
- 支持分组特殊倍率和分组特殊可用分组
- 支持自动分组 `auto` 和默认分组策略
- 支持模型组、标签组、端点组等预填分组复用能力

这一章的核心价值表达是：平台不仅能接模型，还能把资源分配、计费和权限策略精细化落到不同用户群体。

### 7. 适用场景

使用业务场景来收束前文能力，例如：

- 企业统一 AI 模型出口
- 团队共享多家模型供应商资源
- 需要多渠道容灾、切换和成本控制的业务平台
- 面向内部部门或外部客户提供统一 API 服务的平台

### 8. 总结

结尾要强化一个判断：

这不是一个仅做 API 转发的工具，而是一个具备统一接入、统一治理和统一运营能力的 AI 平台。

## Tone

最终文案采用产品化、汇报友好的表达方式。

语气要求：

- 稳定、清晰、专业
- 强调价值，但不过度营销
- 强调平台治理能力，不堆砌技术术语
- 用完整语句表达业务意义，不使用开发备注式短句

避免：

- “支持 XX 字段”
- “接口参数如下”
- “点击按钮后即可”
- “未来可扩展为”

推荐：

- “平台支持统一管理……”
- “管理员可以集中配置……”
- “适合用于……”
- “能够帮助团队实现……”

## Content Boundaries

成稿时必须遵守以下边界：

- 可以总结现有网站和代码中的产品能力
- 可以提炼这些能力带来的管理价值和业务价值
- 不能编造当前未在页面或接口中落地的功能
- 不能写成企业专属、审计合规、SLA 等未经确认的能力承诺
- 不能把内部实现细节当成对外交付能力卖点

## Success Criteria

当最终 Markdown 满足以下条件时，视为设计完成：

- 非技术读者可以在短时间内理解平台定位
- 客户能看出管理员后台是该平台的重要竞争力
- 系统设置、渠道管理、分组管理三个重点章节清晰且有说服力
- 文案读起来像对外产品介绍，而不是后台操作文档
- 文案内容全部能够被当前页面、导航和代码能力支撑
