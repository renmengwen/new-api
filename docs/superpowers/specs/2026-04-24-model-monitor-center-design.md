# 模型监控中心设计

## 背景

现有“系统设置 - 运营设置 - 监控设置”中的定时测试能力以通道为基本单位。它会遍历通道，并为每个通道选择一个测试模型，用于判断通道配置、密钥、上游可用性是否健康。

新的模型监控中心面向管理员回答另一个问题：系统中每个模型在其所属渠道上的实际可用状态如何。页面以模型为主展示，但需要明确展示模型所属渠道，便于管理员快速看出一个模型在哪些渠道成功、失败或超时。

## 目标

- 将监控设置从运营设置中独立出来，形成“模型监控中心”页面。
- 主表以模型为基本展示单位，一行一个模型。
- 每个模型显示所属渠道摘要，渠道状态通过 Tag 颜色表达。
- 定时测试时，测试单位为“模型 + 所属渠道”：如果 A、B 两个渠道都包含同一个模型，则该模型会分别在 A、B 上测试。
- 支持为每个模型配置最长响应时间。
- 支持为每个模型关闭定时测试。
- 失败通知以模型维度汇总发送给管理员，避免逐渠道逐模型发送造成噪音。

## 非目标

- 不做单独的通道诊断页面。
- 不把主表改成一行一个“模型 + 渠道”的扁平列表。
- 不由模型监控直接自动禁用通道；通道禁用仍由现有通道监控和自动禁用逻辑负责。
- 不默认高频测试高成本、视频、图像、realtime 等风险模型。

## 页面结构

页面名称：模型监控中心。

顶部为全局设置区：

- 启用模型定时测试。
- 测试间隔。
- 默认最长响应时间。
- 并发数。
- 失败通知策略。
- 默认跳过规则。

中间为模型列表主表：

| 字段 | 说明 |
| --- | --- |
| 模型 | 模型名称，一行一个模型 |
| 所属渠道 | 展示前几个渠道 Tag，超出显示 `+N` |
| 状态 | 正常、部分异常、不可用、跳过、无可测渠道 |
| 成功/失败 | 该模型在所属渠道上的测试成功数和失败数 |
| 最长响应 | 模型级可编辑阈值 |
| 最后测试 | 最近一次测试时间 |
| 定时测试 | 模型级开关 |

模型行支持展开，展开后显示该模型在各所属渠道上的测试明细：

| 字段 | 说明 |
| --- | --- |
| 渠道 | 渠道名称 |
| 渠道状态 | 启用、自动禁用、手动禁用等 |
| 测试结果 | 成功、失败、超时、跳过 |
| 响应时间 | 本次测试耗时 |
| 错误摘要 | 上游错误或本地错误摘要 |

右侧或顶部摘要区展示本轮测试统计：

- 模型总数。
- 参与测试模型数。
- 正常模型数。
- 部分异常模型数。
- 不可用模型数。
- 跳过模型数。

参考线框：`docs/superpowers/mockups/model-monitor-center-wireframe.html`。

## 状态规则

模型状态按该模型的所属渠道测试结果聚合：

- 正常：所有参与测试的所属渠道都成功。
- 部分异常：至少一个所属渠道成功，且至少一个所属渠道失败或超时。
- 不可用：所有参与测试的所属渠道都失败或超时。
- 跳过：模型关闭定时测试。
- 无可测渠道：模型存在，但当前没有启用且可测试的所属渠道。

所属渠道 Tag 颜色：

- 绿色：该模型在此渠道测试成功。
- 红色：该模型在此渠道测试失败。
- 黄色：该模型在此渠道超时。
- 灰色：跳过、渠道未启用或不可测。

## 测试范围

定时任务先构建模型到渠道的映射：

1. 读取启用或可展示的渠道。
2. 解析每个渠道的模型列表。
3. 按模型名聚合所属渠道。
4. 根据模型级配置过滤关闭测试的模型。
5. 根据默认跳过规则过滤高风险模型。
6. 对每个剩余模型，测试其每个所属渠道。

如果两个渠道 A 和 B 都包含 `Claude-opus-4-7`，则定时测试会分别执行：

- `Claude-opus-4-7` on A。
- `Claude-opus-4-7` on B。

主表仍只展示一行 `Claude-opus-4-7`，所属渠道列展示 A、B，展开后展示 A、B 的明细结果。

## 配置存储

建议新增 `model_monitor_setting` 配置，保存全局配置和模型级覆盖配置。

示例：

```json
{
  "enabled": true,
  "interval_minutes": 10,
  "default_timeout_seconds": 30,
  "max_concurrency": 3,
  "notify_enabled": true,
  "notify_on_partial_failure": false,
  "excluded_model_patterns": [
    "*image*",
    "*video*",
    "*realtime*"
  ],
  "model_overrides": {
    "Claude-opus-4-7": {
      "enabled": true,
      "timeout_seconds": 45
    },
    "gpt-image-1": {
      "enabled": false,
      "timeout_seconds": 120
    }
  }
}
```

测试结果建议与配置分离，避免高频写入配置表。可以新增模型监控状态表，用于保存每个“模型 + 渠道”的最近测试结果。

核心字段：

- model_name。
- channel_id。
- channel_name。
- status。
- response_time_ms。
- error_message。
- tested_at。
- consecutive_failures。

## 通知策略

默认只在模型不可用时发送通知。

可选开启“部分异常也通知”。通知内容按模型汇总：

- 本轮测试摘要。
- 不可用模型列表。
- 部分异常模型列表。
- 每个模型展示失败渠道数量、成功渠道数量、错误摘要。
- 详情过多时只展示前 N 条，其余省略。

通知对象：

- 默认通知超级管理员。
- 后续可扩展为通知所有开启相关通知的管理员。

## 与现有通道监控的关系

现有通道监控继续保留，用于判断通道健康和触发自动禁用。

模型监控只负责展示模型维度的可用性和通知，不直接修改通道状态。这样可以避免因为某个模型不支持、模型映射错误或临时限流而误禁用整个通道。

## 验证要点

- 构建模型到渠道映射时，A、B 两个渠道同时包含同一模型时，主表只出现一个模型行。
- 展开模型行时，能看到 A、B 两个渠道的独立测试结果。
- 模型级最长响应时间覆盖全局默认值。
- 模型关闭定时测试后不会参与定时任务。
- 失败通知按模型聚合，不逐渠道发送多封邮件。
- 现有通道自动禁用逻辑不受影响。
