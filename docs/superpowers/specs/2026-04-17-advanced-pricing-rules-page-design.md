# 高级定价规则页面设计

## 1. 背景

当前系统的模型价格配置入口为：

- `系统设置 -> 分组与模型定价设置 -> 价格设置`

现有价格设置页适合表达以下两类简单定价：

- 固定按量价格
- 固定按次价格

其核心特点是：

- 一个模型对应一组固定价格
- 前端编辑态是固定字段表单
- 后端存储是 `model -> float` 的平面映射

这套设计适用于：

- OpenAI / Claude / 大多数固定输入输出价模型
- 图片生成等固定按次收费模型

但不适用于以下复杂定价场景：

- Gemini 的上下文长度分层
- 火山文本模型的输入长度 / 输出长度条件组合
- 火山视频模型的任务型矩阵计费

因此需要新增专门的高级规则配置入口，同时保留现有价格设置页。

---

## 2. 设计目标

本设计目标如下：

1. 保留现有价格设置页，继续承担简单固定价格配置
2. 新增“高级定价规则”页面，用于配置复杂文本分段规则和媒体任务规则
3. 避免两个页面配置割裂，确保运营能明确知道某模型当前采用哪种计费模式
4. 保证运行时计费、使用日志、任务日志、额度流水等说明口径一致

---

## 3. 页面定位

新增页面路径建议为：

- `系统设置 -> 分组与模型定价设置 -> 高级定价规则`

页面定位：

- 它不是替代现有 `价格设置` 页
- 它是现有价格设置的补充页
- 专门用于表达现有固定价格页无法表示的复杂计费规则

---

## 4. 两页职责边界

### 4.1 价格设置页

继续负责：

- 固定按量价格
- 固定按次价格

适用模型：

- OpenAI / Claude / 普通文本模型
- 固定按次收费的图片模型
- 其他可用单组固定价格表达的模型

### 4.2 高级定价规则页

只负责当前价格设置页无法表达的复杂规则：

- 文本分段规则
- 媒体任务规则

不重复承载：

- 固定按次价格
- 固定按量价格

即：

- “按次计费”继续保留在现有 `价格设置` 页
- “复杂分段 / 条件 / 视频任务规则”进入新页

---

## 5. 单模型单生效模式

### 5.1 核心规则

每个模型在同一时刻必须有且仅有一个“当前生效计费模式”。

建议支持的模式：

- `固定按量`
- `固定按次`
- `高级规则`

### 5.2 配置保留规则

固定价格配置与高级规则配置可以同时存在，但运行时只使用当前被选中的模式。

即：

- 可以同时保留固定价格
- 也可以同时保留高级规则
- 但系统结算只按当前生效模式执行

### 5.3 为什么采用该方案

这样做的好处：

- 避免两页同时生效造成冲突
- 运营可以清楚看到某模型当前按什么方式计费
- 切换模式时无需删除另一套配置
- 方便灰度切换与快速回滚

不建议的方式：

- 依赖运行时隐式优先级自动选择
- 切模式时删除未生效配置

---

## 6. 现有价格设置页改造

为避免两个页面割裂，建议在现有 `价格设置` 列表中增加一列：

- `计费模式`

显示值：

- 固定按量
- 固定按次
- 高级规则

同时建议增加操作入口：

- 编辑固定价格
- 编辑高级规则
- 切换当前生效模式

### 6.1 列表行为

在价格设置页中：

- 运营可直接看到每个模型当前采用哪种计费模式
- 可从该页跳转到高级规则页编辑复杂规则
- 可在该页切换某模型当前生效模式

### 6.2 右侧详情区行为

当前价格设置页右侧详情区继续仅负责固定价格配置，不直接嵌入复杂规则编辑器。

理由：

- 现有页保持简单
- 避免在固定价格表单里塞入规则矩阵
- 复杂规则编辑器应单独放在高级规则页

---

## 7. 高级定价规则页的规则类型

高级定价规则页建议分为两类规则类型，而不是做一个万能规则编辑器。

### 7.1 文本分段规则

适用模型：

- Gemini 上下文长度分层模型
- 火山文本模型
- 其他按 token 区间 / 条件组合定价的文本模型

支持维度：

- 输入长度区间
- 输出长度区间
- 服务层级
- 缓存相关价格
- 非 token 计费单位：`per_second`、`per_image`
- 图片输出档位：`image_size_tier`
- 工具调用语义：`grounding / web_search / google_search` 归一按 `google_search` 命中
- 工具附加费口径：`free_quota / tool_overage_price`，即免费额度后的超额部分按工具超额单价结算
- 批处理 / 优先级 / 其他文本类条件

典型场景：

- `输入长度 [0,32] / (32,128] / (128,256]`
- 某些模型再按输出长度进一步分层
- Gemini Live 按秒计费
- Gemini 图片生成按张计费，并按 `image_size_tier` 区分档位
- grounding / web_search 请求按统一的 `google_search` 语义命中工具调用规则

### 7.2 媒体任务规则

适用模型：

- 火山视频生成模型
- 以后类似视频 / 复杂任务型模型

支持维度：

- 在线 / 离线
- 是否包含输入视频
- 是否包含音频
- 分辨率
- 宽高比
- 输出时长
- Draft 折算
- 最低 token 阈值
- 最终按任务 usage / total_tokens 结算

该类规则本质上不是普通文本 token 分段，而是任务型矩阵规则。

---

## 8. 页面结构建议

高级定价规则页建议包含以下结构：

### 8.1 顶部说明区

展示：

- 页面用途
- 与现有价格设置页的关系
- 当前模型当前生效模式

### 8.2 模型列表区

用于选择当前编辑模型。

建议字段：

- 模型名称
- 当前计费模式
- 是否存在固定价格配置
- 是否存在高级规则配置
- 规则类型

### 8.3 规则编辑区

根据规则类型切换不同编辑器：

- 文本分段规则编辑器
- 媒体任务规则编辑器

不做统一万能表单。

### 8.4 预览区

显示保存后该模型会写入哪些规则数据，以及当前模型的规则摘要。

预览区需要直接按命中的真实 `billing_unit` 展示公式与日志摘要：

- `per_second` 按秒数展示
- `per_image` 按图片张数展示
- `per_1000_calls` 按 `tool_usage_count`、`free_quota` 与超额单价展示

不再统一伪装成 token 公式。

---

## 9. 运行时结算规则

运行时只按模型的“当前生效计费模式”执行。

### 9.1 固定按量

读取现有：

- `ModelRatio`
- `CompletionRatio`
- `CacheRatio`
- `CreateCacheRatio`
- `ImageRatio`
- `AudioRatio`
- `AudioCompletionRatio`

### 9.2 固定按次

读取现有：

- `ModelPrice`

### 9.3 高级规则

读取：

- 该模型已保存的高级规则配置

并根据请求/任务实际条件命中对应规则后完成结算。

当前已落地的高级规则运行时能力包括：

- 文本高级规则已可按 `per_second`、`per_image`、`per_1000_calls` 结算
- 文本高级规则命中时已可采集并写入 `image_size_tier`
- `grounding`、`web_search`、`google_search` 在运行时统一按 `google_search` 语义参与匹配与计费
- 工具调用类规则已支持 `free_quota / tool_overage_price` 口径；当前超额单价仍由命中规则中的单价字段承载，而不是单独拆出独立配置页字段

---

## 10. 使用日志与账务说明要求

### 10.1 硬性要求

命中高级规则的请求，其：

- 使用日志详情
- 使用日志计费过程
- 任务日志说明
- 额度流水原因/备注（至少是可追溯口径）

必须按该次请求实际命中的规则生成说明。

不得继续沿用固定按量模板或固定按次模板。

### 10.2 使用日志说明口径

如果当前模型生效模式为：

- 固定按量：按现有固定按量说明生成
- 固定按次：按现有固定按次说明生成
- 高级规则：按命中规则生成说明

其中高级规则说明必须按真实 `billing_unit` 输出：

- `per_second`：说明实际计费秒数与按秒公式
- `per_image`：说明实际计费图片数与按张公式
- `per_1000_calls`：说明 `tool_usage_count`、`free_quota`、`tool_overage_price`、超额阈值与超额部分公式

不得把上述三类口径回退成统一的 token 公式。

### 10.3 示例

文本分段规则命中后，使用日志应能说明：

- 规则类型
- 命中条件
- 命中的输入价格 / 输出价格
- 最终计费公式

媒体任务规则命中后，使用日志应能说明：

- 规则类型
- 命中条件
- 任务 usage / total_tokens
- 单价来源
- 最终计费公式

如果命中的是工具调用附加费规则，还应能说明：

- `tool_usage_type` 已按 `google_search` 语义归一
- `free_quota` 以内不计费、超出后如何折算
- 超额部分使用的 `tool_overage_price` 来源

---

## 11. 与统计页面的关系

高级规则生效后：

- 运营分析台
- 使用日志
- 额度流水
- 任务日志

必须读取一致的最终扣费结果。

其中：

- 运营分析台继续展示最终费用
- 不要求运营分析台直接暴露复杂规则细节
- 但费用口径必须与高级规则结算结果保持一致

---

## 12. 兼容性原则

### 12.1 老模型兼容

未配置高级规则的模型继续按现有逻辑工作，不受影响。

### 12.2 已有价格兼容

现有：

- 固定按量价格
- 固定按次价格

不需要迁移才能继续使用。

### 12.3 渐进启用

高级规则可先只覆盖：

- Gemini 分段模型
- 火山文本分段模型
- 火山视频任务模型

后续再扩展到更多模型。

当前仍未完整解决的能力边界：

- 同一模型内按输入模态与输出模态分别给出完全独立价表，仍属于后续增强
- 部分 Live / Image 场景更复杂的输出档位、按秒细分与多段阶梯价，当前仅覆盖已落地的基础 `per_second` / `per_image` 口径
- 长期 `cache_storage` 的按小时或更长周期计费口径尚未在运行时完整结算
- 完整 grounding 免费层、RPD 与更外层搜索生态口径仍未完全纳入统一高级规则体系

---

## 13. 产品建议

建议按以下策略落地：

1. 保留现有 `价格设置` 页不动作为主入口
2. 在其列表中新增 `计费模式` 列
3. 新增 `高级定价规则` 页
4. 采用“单模型单生效模式”原则
5. 高级规则页内按规则类型拆分，而不是做统一万能表单

---

## 14. 结论

本设计的最终结论是：

- 不继续把复杂规则硬塞进现有价格设置页
- 不新建一个完全替代现有价格设置的新价格中心
- 最优方案是：
  - 保留现有 `价格设置`
  - 新增 `高级定价规则`
  - 通过 `计费模式` 列连接两页
  - 让每个模型在任一时刻只采用一种生效计费模式

该方案兼顾了：

- 简单模型的易用性
- 复杂规则的表达能力
- 运行时口径一致性
- 日志与账务可解释性
