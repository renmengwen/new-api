# 正式环境双 new-api 节点负载均衡部署说明

本文档用于记录本次正式环境从单台 `new-api` 应用服务器扩容为两台 `new-api` 应用服务器，并通过 CDN/负载均衡对外提供服务的落地方案。

## 目标架构

```text
用户
  ↓
CDN
  ↓
负载均衡
  ↓
new-api 应用服务器 1（web1，master）
new-api 应用服务器 2（web2，slave）
  ↓
正式数据库服务器
正式 Redis 服务器
```

数据库和 Redis 已经单独部署，本次扩容不迁移数据库和 Redis。

两台 `new-api` 应用服务器连接同一套正式数据库和同一套正式 Redis。

## 服务器角色

| 角色 | 说明 |
| --- | --- |
| web1 | 原正式 `new-api` 应用服务器，主节点 |
| web2 | 新增 `new-api` 应用服务器，从节点 |
| DB | 正式数据库服务器，不在本次应用扩容范围内 |
| Redis | 正式 Redis 服务器，不在本次应用扩容范围内 |
| CDN/LB | 负责用户入口、HTTPS 证书、负载均衡和健康检查 |

## new-api 部署目录

两台应用服务器部署目录保持一致：

```
/root/new-api/
  .env
  docker-compose.yml
```

正式环境不需要在服务器上放完整源码。GitHub Actions 会构建 Docker 镜像并推送到 GHCR，应用服务器只需要拉取镜像并重启容器。

## 环境变量要求

两台应用服务器的 `.env` 必须连接同一套正式数据库和 Redis。

web1：

```env
TZ=Asia/Shanghai
PORT=3000
SQL_DSN=正式数据库连接
REDIS_CONN_STRING=正式Redis连接
NODE_TYPE=master
SESSION_SECRET=两台完全一致的随机密钥
CRYPTO_SECRET=两台完全一致的随机密钥
```

web2：

```env
TZ=Asia/Shanghai
PORT=3000
SQL_DSN=和web1完全一致
REDIS_CONN_STRING=和web1完全一致
NODE_TYPE=slave
SESSION_SECRET=和web1完全一致
CRYPTO_SECRET=和web1完全一致
```

说明：

- `NODE_TYPE` 不写时默认等同于 `master`。
- 多节点部署建议 web1 明确写 `NODE_TYPE=master`，web2 明确写 `NODE_TYPE=slave`。
- `SESSION_SECRET` 必须两台一致，否则用户登录状态可能不一致。
- `CRYPTO_SECRET` 必须两台一致，否则 Redis 或加密数据可能无法正确解密。
- 修改 `SESSION_SECRET` 后，已登录用户可能需要重新登录一次。

## GitHub Actions 自动部署

正式环境部署由 `.github/workflows/deploy.yml` 负责。

当前正式部署流程：

```
push main
  ↓
构建 Docker 镜像 ghcr.io/renmengwen/new-api:latest
  ↓
部署 web1
  ↓
部署 web2
```

GitHub Secrets 约定：

| Secret | 说明 |
| --- | --- |
| `DEPLOY_HOST` | web1 服务器 IP |
| `DEPLOY_PORT` | web1 SSH 端口 |
| `DEPLOY_HOST_2` | web2 服务器 IP |
| `DEPLOY_PORT_2` | web2 SSH 端口，当前为 `57891` |
| `DEPLOY_SSH_KEY` | GitHub Actions 使用的 SSH 私钥 |

两台服务器都需要在 `/root/.ssh/authorized_keys` 中配置 `DEPLOY_SSH_KEY` 对应的公钥。


GitHub Actions 部署时会在两台应用服务器执行：

```bash
cd /root/new-api
docker compose pull new-api
docker compose up -d
docker image prune -f
```

## 日志和数据存储

业务日志不是写在本地 log 文件中。

| 类型 | 存储位置 |
| --- | --- |
| 使用日志 | 数据库 `logs` 表 |
| 额度流水 | 数据库 `quota_ledgers` 表 |
| 审计日志 | 数据库 `admin_audit_logs` 表 |
| 程序运行日志 | 各应用服务器本地日志或 Docker 日志 |

当前 `.env` 未配置 `LOG_SQL_DSN` 时，使用日志和主业务数据使用同一个正式数据库。

如果后续希望把使用日志单独拆到日志数据库，可以增加：

```env
LOG_SQL_DSN=日志数据库连接
```

注意：

- `LOG_SQL_DSN` 只影响使用日志 `logs` 表。
- 额度流水和审计日志仍建议保留在主数据库。
- 两台应用服务器必须配置相同的 `LOG_SQL_DSN`。
- master 节点负责日志库表结构迁移。

## 异步导出文件注意事项

使用日志、额度流水、审计日志的数据本身在数据库中，但异步导出的 Excel 文件会落到应用服务器本地 `EXPORT_DIR`。

多节点下需要注意：

- 异步导出任务只应由 master 节点执行。
- 如果导出文件生成在 web1，本地文件不一定存在于 web2。
- 如果下载导出文件的请求被负载均衡转发到 web2，可能出现文件不存在。

建议二选一：

1. 将 `EXPORT_DIR` 配置为两台服务器共享目录。
2. 将导出文件下载相关请求固定转发到 web1。

## 上线验证步骤

web1 本机验证：

```bash
cd /root/new-api
docker compose ps
curl http://127.0.0.1:3000/api/status
```

web2 本机验证：

```bash
cd /root/new-api
docker compose ps
curl http://127.0.0.1:3000/api/status
```

负载均衡侧验证：

```bash
curl http://web1内网IP:3000/api/status
curl http://web2内网IP:3000/api/status
```

GitHub Actions 验证：

```text
Actions -> Deploy -> deploy_prod
```

需要确认以下两个步骤均成功：

```text
Deploy production web1
Deploy production web2
```

业务验证：

- 登录后台。
- 创建或查询 token。
- 调用 `/v1/chat/completions`。
- 查看使用日志。
- 查看额度流水。
- 查看审计日志。
- 如启用异步导出，验证导出文件下载。

## 回滚方案

如果 web2 接入负载均衡后异常：

1. 在负载均衡中摘除 web2。
2. 保留 web1 正常对外服务。
3. 在 web2 上检查容器状态：

```bash
cd /root/new-api
docker compose ps
docker compose logs --tail=200 new-api
curl http://127.0.0.1:3000/api/status
```

如果 GitHub Actions 部署异常：

1. 先确认 web1 是否已成功部署。
2. 如果 web2 部署失败，先不要把 web2 加入负载均衡。
3. 修复 SSH、Docker 或配置问题后重新运行 GitHub Actions。

如果新版本业务异常：

1. 从负载均衡中临时摘除异常节点。
2. 使用上一版镜像或上一版代码重新触发部署。
3. 确认 `/api/status` 和核心业务接口正常后再恢复流量。
